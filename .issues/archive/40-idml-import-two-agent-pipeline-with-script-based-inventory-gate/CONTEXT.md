# Context — Two-agent pipeline + script-based inventory gate

## Problem

The IDML→Scribus import converter (`tools/idml_to_dsl.py`) has become a dumping ground for per-template visual-fidelity heuristics. A 30-iteration session on the 26-03 Leporello template produced a 916-line diff in which roughly 200 lines were genuine converter improvements and 700 lines were template-specific fixes that should have lived outside the converter. Every visual fix had to be correct for *every conceivable IDML* — fixing one template's photo crop broke another template's icon rendering.

There is no deterministic gate that says "this scaffold is structurally complete." Today's signals (`drift_p1`, visual diff) hide regressions of individual elements behind aggregate pixel deltas, and they can't tell the difference between "missing word" and "word in slightly wrong place".

## Decisions locked in for this issue

| Decision | Choice | Rationale |
| :--- | :--- | :--- |
| Scope | **Full plan, one PR** | User explicit. Inventory + skill split + SOP rewrite ship together. |
| Anchor template | **26-03 Leporello z-Falz** | Already tuned, known-good baseline.pdf, 444 words, motivated this issue. |
| Pipeline | **Run through execute** | issue-executor agent dispatched with worktree isolation. |
| Inventory comparison | **Deterministic script, NOT LLM** | User explicit: "script comparison if all elements exist". |
| Stage 2 forbidden paths | `tools/idml_to_dsl.py`, `tools/sla_lib/**`, shared converter code | Tuning agent must not regress other templates. |
| Stage 2 permitted paths | `templates/<slug>/build.py`, `inject.yml`, `meta.yml::brand_overrides`, `TOLERANCES.yml` | Per-template only. |

### Existing artifacts modified

### Schema (canonical)

`templates/<slug>/SCAFFOLD_INVENTORY.yml` keys:
- `text_runs` — counts by paragraph style, set-equality on (text, font, fontsize)
- `frames` — text_frames, image_frames, polygon_frames, group_frames; each row has IDML position, build.py position, sla_pageobject_present, sla_storytext_runs OR sla_pfile_present, pdf_image_present
- `paragraph_styles` — IDML name ↔ build.py name ↔ sla_pstyle_present
- `colors` — IDML name + cmyk ↔ build.py extra_color flag ↔ sla_color_present
- `assets` — basename, on_disk, classified (external|embedded), referenced_from_frames
- `words` — baseline_pdf_count, preview_pdf_count, missing_from_preview, extra_in_preview

### Gate behavior

Stage 1 (scaffold) **blocks** if any of:
- IDML `<CharacterStyleRange>` text missing from build.py
- IDML frame `Self` ID without corresponding build.py emit
- IDML paragraph style without `add_para_style` call
- IDML Link basename not on disk
- `python3 build.py` exits non-zero
- `render_pipeline.py` exits non-zero
- preview.pdf word count differs from build.py text-content character count by > 5%

Stage 2 (tuning) **blocks** any iteration that:
- Decreases any `count` field in inventory_diff
- Removes an `anname` from any frame list
- Drops a word from `preview_pdf_words` present in the prior iteration

## Open questions for research phase

1. **build.py walker — AST vs importlib?** AST is safer (no side effects), importlib is more accurate (resolves brand_overrides, conditionals). Recommend AST-first with importlib fallback for ambiguous calls.
2. **PDF word matching — by string or position?** Position-aware matching catches "word in wrong frame" but requires pdfplumber + tolerance tuning. String-only is simpler. Schema supports both — start with string-only, expand later.
3. **Composite-AI assets in inventory** — count once or per-frame? They're one physical asset but referenced from N frames with different offsets. Treat as one asset with multiple `referenced_from_frames` entries.
4. **Where does `inject.yml` live in the schema?** Stage 2 edits it heavily for text overrides; inventory should distinguish "build.py text" from "inject.yml override text".
5. **TOLERANCES.yml format** — proposed: `{element_anname: {axis: pixels, justification: "..."}}`. Validates against gate.

## Non-goals for this PR

- Migrating existing tuned templates (26-03 Leporello stays tuned, just gains SCAFFOLD_INVENTORY.yml retroactively)
- Replacing `drift_p1` — it stays as a secondary signal
- Rewriting `tools/idml_to_dsl.py` — only minor changes to emit the inventory metadata
- Slack/CI integration of the inventory gate — that's follow-up
- Composite-AI splitter overhaul — gate just records what splitter produced

## Calibration plan

26-03 Leporello has known-good rendering. Procedure:
1. Run `inventory_extract.py` against current state → expected snapshot
2. Hand-verify that every counted element is real (no false positives)
3. Hand-verify zero missing elements (no false negatives) by spot-checking baseline.pdf
4. Lock the snapshot as `templates/26-03-.../SCAFFOLD_INVENTORY.yml`
5. Re-run on other tuned templates if any exist; record results
6. Mutation testing: deliberately break build.py (remove a word, swap a color) → confirm gate catches it

## Risks

- **AST walker brittleness** — Build.py uses helpers, list comprehensions, conditional branches. Importlib gives ground truth but executes side effects. Mitigation: import build.py with mocked `add_*` recorders.
- **Frame ID drift** — IDML `Self` IDs are not stable across IDML re-exports. Mitigation: index by (kind, mm_position±0.5) tuple as secondary key.
- **Word ordering in PDF** — pdftotext can re-order across columns. Mitigation: word-set comparison, position only as tiebreaker.
- **Test churn** — every existing template's build.py needs an inventory baseline before gate enforcement. Mitigation: gate has `--baseline-only` mode that writes the snapshot instead of comparing.
