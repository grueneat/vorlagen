## 0. TL;DR for the planner

- The 5 new templates already use a clean `if path.exists()` conditional-inject
  pattern via `pack_inline_image()` (see `templates/kandidat-falzflyer-din-lang/build.py:207-220`,
  `templates/wahltag-tueranhaenger/build.py:309-322`, etc.). Migration on those 5
  is a 2-line diff per slot: replace `(HERE/"samples"/"x.jpg").read_bytes()` with
  `library.load("x")` (returns bytes or `None`).
- The 3 production templates (postkarte / plakat / zeitung) were generated by
  `tools/sla_to_dsl.py` and carry **inline_image_data verbatim** copied from the
  upstream original SLAs. There is **no `samples/` folder** for postkarte and
  plakat (postkarte has `klimaschutz.json` / `wohnen.json` only — these are not
  images), and zeitung has none at all. Their image-bearing PAGEOBJECTs are
  either (a) already filled with the original's inline bytes, or (b) emit
  `image=''` (an empty PFILE attribute, no inline data). Adding inline data to
  the `image=''` slots would break round-trip — the original SLA does not have
  `isInlineImage="1"` on those frames. This is exactly the situation D3
  (`<slug>-preview.sla`) is designed to handle.
- Total empty image slots: postkarte=8 (only 1 is gallery-relevant — a 84×127mm
  hero on page 0, line 86); plakat=3 (1 large 594×414mm hero on line 151);
  zeitung=20 (Cover + Foto-Spread + several theme blocks). Many "empty" slots
  in postkarte are 2.5×2.5mm address dots and small icons that the original
  designer left empty intentionally — those should stay empty.
- The library helper belongs at `tools/sla_lib/builder/library.py` (alongside
  `primitives.py`, `blocks.py`, `ci.py`). Reasons in §4 below. Templates already
  import from `sla_lib.builder` — adding `library` there matches the existing
  surface.
- `crop_for_frame()` is a thin Pillow center-crop + JPEG re-encode — pattern
  matches `circular_mask()` in `tools/qr_gen.py:157-181`. Pillow 12.2.0 is
  pinned by Dockerfile + CI workflow; deterministic JPEG q=80 + `optimize=True`
  reproduces qr_gen's PNG-determinism story (verified for #11 as
  `test_qr_byte_determinism`).
- CI doesn't render the 8 templates — only the smoke fixtures rebuild from
  build.py. CI runs `check-stale-previews` which compares
  `SHA256(template.sla)` against `meta.yml::previews_for_sla`. This means **the
  preview-SLA pattern needs zero CI changes** as long as we hash `template.sla`
  (the clean one), not `template-preview.sla`.

**Pillow's PNG-determinism behaviour (key facts for crop_for_frame):**
- `Pillow==12.2.0` pinned (Dockerfile.claude + workflow line 55).
- `optimize=True` strips creation timestamps, software tags, etc.
- JPEG q=80 with `optimize=True` is byte-identical across runs **when the input
  PIL.Image is byte-identical** (proven by qr_gen test suite).
- Center-crop on identical input → identical output.

### 1.3 `tools/sla_lib/builder/primitives.py` — `pack_inline_image`

```
pack_inline_image(image_bytes: bytes, ext: str) -> tuple[str, str]
    primitives.py:750-761
```

- qCompress format: `base64( 4-byte big-endian uncompressed-length prefix + zlib_compress(bytes, 6) )`.
- Returns `(qcompressed_b64, ext)`. Pass to ImageFrame as
  `inline_image_data=...`, `inline_image_ext=ext`.
- All 5 new templates use this verbatim. Library `LibraryImage.bytes` is the
  raw image bytes — caller still calls `pack_inline_image(img.bytes, "jpg")`.
- ImageFrame supports `inline_image_data=None` / `inline_image_ext=None` for
  empty slots (lines 786-790, 793-794, 821-831). The conditional-inject pattern
  becomes unconditional inside `build_preview()` since the library is
  guaranteed to have the image (fail-fast on unknown ID per D2).

### 1.6 `tools/sla_diff.py` — round-trip path

| Symbol | Line | Purpose |
|---|---|---|
| `_decode_inline_image_sha(b64)` | 500-515 | Decodes qCompress, returns SHA256 of inner bytes. |
| `_chain_topology` / `_chain_hash` | 518-549 | Chain detection — unrelated to image diff. |
| `inline-image-content-mismatch` | 774-776 | Both sides have `isInlineImage="1"` but bytes differ → **WARNING**. |
| `inline-vs-sidecar-image` | 783-791 | One side inline, the other PFILE → **CRITICAL**. |

**Critical implication for production templates:** the original SLAs have
PAGEOBJECTs with empty PFILE (`image=''`, no `isInlineImage`). If a build.py
that aspires to fill those slots emits `isInlineImage="1"` with new ImageData,
sla_diff fires `inline-vs-sidecar-image` (CRITICAL) and CI fails. **The clean
`template.sla` MUST keep those slots empty** — exactly the constraint that D3
enforces by routing demo content through a separate `<slug>-preview.sla`.

Both are 15-line shims that exec the corresponding `tools/*.py`. No DSL surface
of their own — modify the underlying tools, the shims pass through.

## 4. Interfaces for the new `tools/sla_lib/builder/library.py`

### 4.1 Module placement decision

Recommended: **`tools/sla_lib/builder/library.py`** — alongside `primitives.py`,
`blocks.py`, `ci.py`. Rationale:

- All 8 templates already import from `sla_lib.builder` (e.g. `from sla_lib.builder import ParaStyle, ImageFrame, pack_inline_image`). Adding `library` to the same import surface is one extra symbol per template — no new sys.path manipulation.
- The library helper is logically a builder concern: it produces bytes that go straight into `pack_inline_image()` and `ImageFrame(inline_image_data=…)`. It's not a CLI tool like `qr_gen.py` or `codex_image_gen.py` that has its own `__main__`. The CLI surface (regen-all) extends `codex_image_gen.py` via the `--library` flag (D6); the library `.py` itself is purely loadtime.
- Module naming: `library` (not `image_library`, not `samples_library`). The shared/sample-images folder is the data; `library` is the access API. Templates read as `from sla_lib.builder import library; portrait = library.load("portrait_maria")` — short and reads naturally.

### 4.4 Library-ID validation hook (CONTEXT "Claude's Discretion")

Per CONTEXT, library-ID-validation can live (a) in `library.load()` itself
(fail-fast at build), or (b) in a separate `tools/check_library.py`.

**Recommendation:** (a) — fail-fast at build time. Reasons:

- Build-time errors surface during `python3 build.py`, which is the first line
  of defense any developer touches. A separate `tools/check_library.py` would
  only be run manually or in CI, missing the in-loop authoring case.
- The cost of (a) is one extra file read (`manifest.yml`) per build.py
  invocation — ~1ms. Negligible.
- (b) is still useful as a CI lint that catches **dead** library entries
  (entries in manifest.yml that no template references). That's a separate
  concern and can be added later as a warning-only tool.

**Implementation:** `library.load(id)` with `optional=False` raises
`LibraryError("unknown library ID: portrait_xxx")` when the ID isn't in
`manifest.yml`. The traceback points to the build.py line that asked. Done.

### 6.1 The brand-validator wildcard (potential issue)

Line 134-136:
```bash
for sla in templates/*/template.sla templates/*/*.sla; do
  [ -f "$sla" ] && python3 tools/check_ci.py "$sla" || true
done
```

The first glob `templates/*/template.sla` matches `template.sla`. The second

matches `template-preview.sla`. The `[ -f "$sla" ]` test passes; `check_ci.py`

check_ci.py invocation per template — negligible.

narrow the second glob to `templates/*/*-original.sla` if the codebase ever

## 8. Gotchas with file moves

### 8.1 git mv vs rename + remove

`git mv old new` is the right tool — preserves blame and reduces diff noise.
The codebase enforces no-claude-attribution but is otherwise relaxed about
commit history.

### 8.2 codex-pipeline-validation note from #11

`tools/codex_image_gen.py:278-281` documents the **stdin=DEVNULL** discovery
from #11: codex 0.128.0 hangs indefinitely on captured stdout without explicit
stdin redirect. The library regenerate flow inherits this fix because it goes
through `generate_image()`. **Do not write a parallel codex invocation that
forgets stdin=DEVNULL.**

### 8.3 Codex output recovery (P-CODEX-PATH)

`tools/codex_image_gen.py:218-269` — codex sometimes writes to
`~/.codex/generated_images/<UUID>.png` instead of the requested path. The
recovery helper handles this, but only when `started_at` is captured before
the codex call. Library regenerate must capture `started_at = time.time()`
and pass it through. Already handled inside `generate_image()` at
codex_image_gen.py:293; library mode just calls generate_image, so OK.

### 8.4 build.py path resolution

The 5 new templates use `HERE / "samples" / "x.jpg"` (template-relative). After
migration, the new convention is **library-relative** (the library helper
resolves `LIBRARY_ROOT = HERE.parents[3] / "shared" / "sample-images"`).

Do NOT mix relative paths in build.py — `library.load("id")` always returns
bytes; build.py never constructs library paths directly. This is the whole
point of D2 (ID-based reference, not path-based).

### 8.6 Manifest mtime triggers regeneration

`image_needs_regen()` regenerates if output is older than the manifest. **When
editing `shared/sample-images/manifest.yml`, ALL library images become regen
candidates** — even if you only added one entry. Use `--force` selectively or
edit prompts surgically. The 5 new images cost ~$0.40 to generate; 12 images
cost ~$1. Acceptable but worth noting in the execution log.

### 8.7 codex_image_gen.py manifest schema mismatch

The current `parse_manifest()` (codex_image_gen.py:75-96) expects `images: [list]`. 
The library manifest is `images: {dict[id]: entry}` per CONTEXT D1.
Library-mode parser is genuinely separate, not a generalisation of the
template-mode parser. Implement as `parse_library_manifest()` rather than
overloading `parse_manifest()`.

### 8.8 SHA gate vs preview-SLA churn

`meta.yml::previews_for_sla` is a 64-char SHA of `template.sla`. The clean
template.sla on production templates **must not change** during this issue —
only `template-preview.sla` is new. So `previews_for_sla` should remain
unchanged on postkarte / plakat / zeitung. If it changes, that's a signal
that build_clean() drifted (likely the `build` → `build_clean` refactor
introduced a non-determinism). Watch this in execution.

For new templates (kandidat-falzflyer, themen-plakat, infostand, tueranhaenger,
wahlaufruf), `template.sla` WILL change because the conditional-inject is
switching from `samples/x.jpg` (which the manifest references) to library
paths. So their `previews_for_sla` SHA must be updated by `bin/render-gallery`
during execution.

### 8.9 sla_lib import cycle

`from sla_lib.builder import library` is fine. `library.py` imports `Path`,
`yaml`, `dataclasses`, `pillow` — no circular dependency with `primitives.py`,
`document.py`, or `blocks.py`. Watch out NOT to import `library` from
`primitives.py` or `__init__.py` aggressively (only re-export the symbols).
The `__init__.py` should add:

```python
from . import library  # exposed as sla_lib.builder.library
```

so callers do `from sla_lib.builder import library; library.load("id")`. This
matches the existing pattern for `from . import blocks` (line 62 of
`tools/sla_lib/builder/__init__.py`).

### 8.10 Pillow JPEG q=80 round-trip determinism

Reading the same JPEG twice with Pillow 12.2.0 → same bytes (no decode quality
loss on read). Saving same `Image` object with same `quality=80, optimize=True`
→ same bytes. **But:** crop+resize+save introduces one decode/encode cycle.
Two consecutive `crop_for_frame(img, w=87, h=24)` calls return identical bytes
because every step is deterministic. Verified pattern in qr_gen tests; same
guarantee applies here.

What can break determinism:
- Pillow version drift across machines (we pin 12.2.0).
- Different `quality` or `optimize` values (always set explicitly in
  `crop_for_frame`).
- Modifying `LibraryImage.bytes` between calls (frozen dataclass prevents this).

## 10. Open questions for the planner

1. **Postkarte production hero (line 86) — does it actually want demo content?**
   The hero is 84×127 mm portrait on the front. The two JSON content slots
   (`klimaschutz.json`, `wohnen.json`) describe the campaign content. If the
   hero is a deliberate empty slot for end-users to fill, demo content might
   actively hurt the gallery (suggests "this slot is filled" when the
   end-user-facing template is empty). **Recommendation:** include in
   preview-SLA but flag clearly in the gallery caption that the hero is an
   empty slot in the downloadable template.

2. **Preview-SLA file naming.** D3 says `<slug>-preview.sla`; render-pipeline
   plan above uses `template-preview.sla` (template-relative). The
   `<slug>-preview.sla` form would put preview SLAs at the repo root which
   conflicts with the existing `*-original.sla` location of upstream
   originals. **Recommendation:** `templates/<slug>/template-preview.sla`
   (template-relative, sibling of `template.sla`). This is what the
   render-pipeline change above already assumes.

3. **What happens if a build_preview() forgets to embed an image?** The
   resulting `template-preview.sla` is identical to `template.sla` (both
   empty slots). `bin/render-gallery` would render an empty preview PDF.
   **Recommendation:** add a tiny smoke test in `tools/sla_lib/tests/` that
   checks: for every template with `build_preview`, the preview-SLA has at
   least one `isInlineImage="1"` PAGEOBJECT that the clean SLA doesn't.
   This is one bash one-liner; easy to add.

4. **Library-manifest schema validation strength.** Permissive vs strict tag
   list (CONTEXT discretion). **Recommendation:** permissive at #13 scope —
   ad-hoc `_validate_manifest_entry` that requires `path:` and `prompt:`,
   warns on missing `tags:` / `license_note:` / `synthetic:` / `size:`,
   accepts any string in `tags:`. Strict tag-namespace validation can land
   in #12 (Spec-System v2) since it overlaps with the brand-constraint DSL.
